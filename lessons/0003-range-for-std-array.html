Kill the loop macros

FOR_EACH_PROJECTILE and FOR_EACH_ENEMY are macros faking a language feature C++ already has: the range-based for loop.

What the macros are hiding

You wrote these to avoid repeating pointer arithmetic:

#define FOR_EACH_ENEMY(enemyPtr, enemyArray)                  \
  for (Enemy* enemyPtr = enemyArray;                          \
       enemyPtr < enemyArray + MAX_ENEMIES; enemyPtr++)

They work, but they carry every macro tax from lesson 2 — no type checking, invisible to tools, and they hardcode MAX_ENEMIES / MAX_PROJECTILES so the loop and the array can silently drift apart. They also force enemy-> arrow syntax everywhere, a tell that you're juggling raw pointers by hand.

The tool: range-based for

C++ can iterate any array or container directly. You already used it once in InitBullets's cousin — here it is on the enemies:

// before
FOR_EACH_ENEMY(enemy, state->enemies) {
  if (!enemy->active) continue;
  enemy->pos = ...;
}

// after — note the & : we want a reference so we can mutate the real element
for (Enemy& enemy : state->enemies) {
  if (!enemy.active) continue;
  enemy.pos = ...;
}

The one rule that matters here: & or no &

for (Enemy& e : ...) binds e to the actual array element — writes stick. for (Enemy e : ...) copies each element into e — writes are thrown away at the end of the iteration. Every one of your loops mutates (sets active, moves pos), so they all need Enemy& / Projectile&. Drop the & and the game silently stops working while still compiling — the classic beginner trap. [cppreference: range-for]

And because enemy is now a reference, not a pointer, member access flips from enemy->active to enemy.active (dot, not arrow).

Why range-for is safe where the macro wasn't

The macro said "go from array to array + MAX_ENEMIES." If the array's real size ever differs from that constant, you walk off the end — undefined behaviour. Range-for asks the array itself for its bounds; loop and storage cannot disagree. That's the structural win, and it sets up the next step.

Optional second step: std::array

Right now bullets/enemies are C arrays in game.h:

Projectile bullets[MAX_PROJECTILES];
Enemy enemies[MAX_ENEMIES];

A C array decays to a bare pointer the moment you pass it to a function — it forgets its own size. std::array is the same fixed-size, zero-overhead storage, but it remembers: it carries .size(), works with range-for cleanly, and refuses to silently decay.

// game.h
#include <array>
std::array<Projectile, MAX_PROJECTILES> bullets;
std::array<Enemy, MAX_ENEMIES> enemies;

Range-for over them is unchanged. state->enemies[i] indexing still works. The payoff grows in lesson 4 when functions take these by reference and keep their size.

Your task feedback loop

  1. Replace every FOR_EACH_PROJECTILE / FOR_EACH_ENEMY use with a range-for. There are ~9 of them — do them one at a time and rebuild often.
  2. For each, decide & vs copy. They all mutate, so they all want &. Convert -> to . as you go.
  3. Mind the nested loop in CheckBulletEnemyCollisions — two range-fors, both by reference.
  4. Delete the two #define FOR_EACH_* blocks. Build. grep -rn FOR_EACH src must be empty.
  5. Then test by playing — this is the trap step. If enemies freeze or bullets don't move, you copied instead of referenced. Find the missing &.
  6. (Optional) Convert the two arrays to std::array and rebuild.
Watch the early-exit loops. UpdateEnemies uses break to bail after finding an edge hit — range-for supports break/continue identically, so those translate one-to-one. No logic change needed.
