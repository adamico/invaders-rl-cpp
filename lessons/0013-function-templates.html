Lesson 13 · Refactor · Templates (new mechanic)

One collision test for every pair

Your two collision functions repeat the same circle test against different types. They have no shared base class — and they don't need one. This is the problem function templates were built for: write the logic once, let the compiler stamp out a version for each type you use it with.

The duplication

CheckCollisionCircles(bullet.pos, bullet.radius, enemy.pos, enemy.radius)   // bullet × enemy
CheckCollisionCircles(state.player.pos, state.player.radius, enemy.pos, enemy.radius)   // player × enemy

Identical except for the types. Player, Enemy, Projectile are three unrelated structs, but they all happen to expose .pos and .radius. You can't write one normal function that takes "anything with pos and radius" — C++ is statically typed, and these share no base class. A template lets you say it anyway.

The tool: a function template

// NEW FILE: include/collision.h
#pragma once
#include "raylib.h"

template <typename A, typename B>
bool overlaps(const A& a, const B& b) {
  return CheckCollisionCircles(a.pos, a.radius, b.pos, b.radius);
}

Read it as a recipe, not a function. template <typename A, typename B> says "A and B are placeholders for types I'll fill in later." When you call overlaps(bullet, enemy), the compiler instantiates a concrete overlaps<Projectile, Enemy> for you; overlaps(player, enemy) generates a second one. You write the logic once; the compiler writes the type-specific copies. [cppreference: function templates]

Compile-time duck typing — and how it differs from your Scene hierarchy

Lesson 6 gave you runtime polymorphism: a Scene* picks the override by the object's actual type while the game runs, via a vtable (a small indirection cost). Templates are compile-time polymorphism: there's no base class, no vtable, no runtime cost — the compiler bakes a separate, fully-typed function for each pair before the program runs. The only "interface" is implicit: any type with .pos and .radius fits. That's "duck typing" — if it has the members the template uses, it works. If it doesn't, you get a compile error at the call site, not a runtime crash. Two tools, two trade-offs: use virtual when you need one container of mixed types chosen at runtime (scenes); use templates when the types are known at compile time and you just want to stop copy-pasting (this).

The gotcha: template definitions live in headers

This is the one mechanical rule that trips everyone. A normal function is declared in a header, defined in a .cpp. A template can't be — the compiler needs the full body visible at every call site to instantiate it for that site's types. If you put overlaps's body in a .cpp, other files see only a declaration, can't instantiate it, and you get a linker error (undefined reference). So: templates go entirely in the header, body and all. That's why collision.h above has no collision.cpp. [isocpp FAQ: why can't I separate template definitions]

Apply it

// scene_gameplay.cpp — add #include "collision.h"

// CheckBulletEnemyCollisions
if (overlaps(bullet, enemy)) {            // was CheckCollisionCircles(bullet.pos, ...)
  PlaySound(state.resources.explosionSound);
  bullet.deactivate();
  state.swarm.deactivate(enemy);
  state.score += enemy.scoreValue;
}

// CheckPlayerEnemyCollisions
if (overlaps(state.player, enemy)) {       // same helper, different type pair
  state.swarm.deactivate(enemy);
  state.player.health = 0;
}

Both call sites now read as intent — "does this bullet overlap that enemy?" — and the circle math exists in exactly one place. Add a new collidable later (an enemy bullet, a powerup) and overlaps already works with it, for free, as long as it carries .pos and .radius.

The other two smells — addressed honestly

Name lies / split check from effect: I floated this last lesson. For an immediate-mode sweep that acts the instant it finds a hit, splitting "detect → return a list → apply later" adds a container and a second loop for no real gain — the effect isn't reused or tested in isolation here. That's overengineering; skip it. A cheaper honesty win is just renaming, e.g. resolveBulletHits over CheckBulletEnemyCollisions (it resolves, not merely checks). Optional. Duplicated if (!active) continue;: still there, still deferred — that one genuinely points at a class template (Pool<T>) and is the next arc, not this one.

Files you'll touch (checklist)

  1. include/collision.h (new) — the overlaps function template, body and all, in the header. No .cpp.
  2. src/scene_gameplay.cpp — #include "collision.h"; replace both CheckCollisionCircles(...) calls with overlaps(a, b).
  3. No Makefile change — header-only, and src/*.cpp is globbed anyway.
  4. Build & play: shooting and player-collision behave exactly as before.
Traps: (1) Don't make a collision.cpp — template in a .cpp = linker "undefined reference." Header only. (2) The template needs a.pos/ a.radius to exist; if you ever pass a type without them, the error appears at the call site on instantiation and can look verbose — read the first line, it names the missing member. (3) const A& — pass by const reference (lesson 4): no copy, and you're only reading.
