Scenes that know themselves

Replace the enum + switch with a Scene hierarchy. This is where inheritance, virtual, the virtual destructor, and unique_ptr + move all earn their place at once.

What's wrong with the switch

Your loop dispatches on an enum:

switch (state.currentScene) {
  case TITLE:    UpdateTitle(&state);    DrawTitle(&state);    break;
  case GAMEPLAY: UpdateGameplay(&state, dt); DrawGameplay(&state); break;
  case GAMEOVER: UpdateGameover(&state); DrawGameover(&state); break;
}

It works, but every new scene (pause menu, settings, level-complete) forces you to edit the enum, edit this switch, edit a second switch if one exists elsewhere, and add a matching pair of free functions. The knowledge of "what scenes exist" is scattered. This violates the open–closed principle: you want to add behaviour by adding a type, not by editing existing control flow. [Core Guidelines C.120: use class hierarchies to represent concepts with inherent hierarchical structure]

The tool: an abstract base class

Define an interface every scene must satisfy, then let each scene be a type:

// scene.h
struct Scene {
  virtual void update(GameState& state, float dt) = 0;   // = 0 : pure virtual, no body
  virtual void draw(const GameState& state) const = 0;
  virtual ~Scene() = default;                            // MUST be virtual — see below
};

Three pieces of new vocabulary, each doing real work:

  • virtual — "resolve which override to call at runtime, based on the object's actual type," not at compile time. This is the dispatch that replaces your switch.
  • = 0 (pure virtual) — "this base provides no implementation; every concrete scene must supply one." It also makes Scene abstract: you can't create a bare Scene, only a TitleScene etc. The compiler enforces the contract.
  • virtual ~Scene() — the one that bites everyone. Read the box.

Why the destructor MUST be virtual

You'll hold scenes through a base pointer: unique_ptr<Scene>. When it deletes a GameplayScene through that Scene*, the runtime needs to know to run ~GameplayScene, not just ~Scene. Without virtual ~Scene(), only the base destructor runs — derived members leak. It's undefined behaviour, and it's silent. Rule: any class meant to be inherited from and deleted via a base pointer needs a virtual destructor. [Core Guidelines C.35]

Each scene becomes a class

Move the body of each Update*/Draw* pair into an override. Your scenes are stateless (they operate on the shared GameState), so the classes are tiny:

// scene_title.h
struct TitleScene : Scene {
  void update(GameState& state, float dt) override;
  void draw(const GameState& state) const override;
};
// scene_title.cpp
void TitleScene::update(GameState& state, float) {   // dt unused here, fine to omit name
  if (IsKeyPressed(KEY_SPACE)) state.currentScene = GAMEPLAY;
}
void TitleScene::draw(const GameState& state) const {
  BeginDrawing();
  ClearBackground(BLACK);
  CenterText(TITLE_TEXT, windowSize.y / 2, 40, WHITE);
  CenterText(SUBTITLE_TEXT, windowSize.y / 2 + 100, 20, WHITE);
  EndDrawing();
}

override is a safety keyword: it tells the compiler "this is meant to replace a base virtual." If your signature doesn't exactly match one (a stray missing const, wrong param type), you get an error instead of silently creating a brand-new unrelated function that never gets called. Always write override.

Ownership & transitions — where unique_ptr + move arrive

Keep your existing currentScene enum as the transition signal (scenes already set it). A factory turns that enum into the matching scene object:

// scene.cpp — returns ownership of a freshly-built scene
std::unique_ptr<Scene> makeScene(GameScene which) {
  switch (which) {
    case TITLE:    return std::make_unique<TitleScene>();
    case GAMEPLAY: return std::make_unique<GameplayScene>();
    case GAMEOVER: return std::make_unique<GameoverScene>();
  }
  return nullptr;
}
"Wait, there's still a switch!" Yes — but exactly one, in one place, doing one job: construction. The per-frame dispatch (the part that grew with every feature) is gone, replaced by a virtual call. Centralising the remaining switch in a factory is the accepted trade-off.

std::unique_ptr<Scene> is an owning pointer with RAII baked in (lesson 1, now automatic): exactly one owner, and it deletes the scene when reassigned or destroyed — through that virtual destructor. You **cannot copy** a unique_ptr (two owners would double-delete — same logic as your GameResources). You can only move it: transfer ownership, leaving the source empty.

// main.cpp
GameState state = {};
InitGameplay(&state);
state.currentScene = TITLE;

std::unique_ptr<Scene> current = makeScene(TITLE);   // ownership moves out of the factory
GameScene shown = TITLE;

while (!WindowShouldClose()) {
  if (state.currentScene != shown) {                 // a scene requested a transition
    current = makeScene(state.currentScene);         // old scene deleted, new one moved in
    shown = state.currentScene;
  }
  current->update(state, GetFrameTime());             // virtual dispatch — no switch
  current->draw(state);
}

Where move actually happens (the bit deferred since lesson 1)

make_unique builds a scene and returns a unique_ptr by value; current = makeScene(...) move-assigns it — the temporary's ownership is transferred into current, and the old scene current held is deleted first. No copy ever happens (it can't — copy is deleted). This is move semantics doing its job invisibly: ownership flows, nothing is duplicated.

Your task feedback loop

  1. Create scene.h with the abstract Scene base (3 virtuals incl. the destructor).
  2. Convert each scene to a struct XScene : Scene with overrides; move the function bodies in. Keep the CenterText helper where it is.
  3. Add makeScene (factory) — the one remaining switch.
  4. Rewrite main's loop to hold a unique_ptr<Scene> and rebuild on transition. #include <memory>.
  5. Delete the old free Update*/Draw* declarations + the dispatch switch.
  6. Build & play: Title → (space) → Gameplay → die → Gameover → (enter) → Title. Every transition must work.
Two traps to expect: (1) forget virtual ~Scene() and the build may pass but leak — add it from the start. (2) GameoverScene::update calls InitGameplay and sets currentScene = TITLE; keep that logic intact so restart still works. Run the full transition cycle, not just the first screen.
