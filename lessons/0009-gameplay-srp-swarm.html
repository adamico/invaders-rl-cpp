Lesson 09 · Refactor · Single Responsibility

The gameplay file knows too much

scene_gameplay.cpp has quietly become a god-file. Before extracting anything, learn to name the responsibilities tangled inside it — then lift exactly one out: the enemy swarm.

Step 1 — diagnose, don't refactor yet

Read your own file and list what it's responsible for. Right now one translation unit owns all of:

  • Entity behaviour — Player::update, Projectile, Enemy methods (already tidy, from lesson 5).
  • Swarm/wave logic — InitEnemies, UpdateEnemies (edge-bounce + descend), DrawEnemies.
  • Cross-entity collision — CheckBulletEnemyCollisions, CheckPlayerEnemyCollisions.
  • Scoring & HUD — score/health text in draw.
  • Scene orchestration — SceneGameplay::update/draw (this is the file's real job).

And look at GameState — four of its fields exist only to serve the swarm:

// game.h — today
typedef struct GameState {
  Player player;
  std::array<Projectile, MAX_PROJECTILES> bullets;
  std::array<Enemy, MAX_ENEMIES> enemies;   // these four
  Vector2 enemyDirection;                   // move
  int enemySpeed;                           // together
  int activeEnemies;                        // always
  bool victory;
  GameResources resources;
  int score;
} GameState;

Four fields that always change together, manipulated only by the *Enemies functions — that's the same "invariant cluster" smell from lessons 5 and 7. The Single Responsibility Principle says a module should have one reason to change. The swarm is its own reason. Give it its own class. [Core Guidelines C.4 / C.8: group data with the operations that maintain its invariants]

Scope discipline (lesson 8 taught us this the hard way)

You could extract all five responsibilities today. Don't. One extraction, build, play, commit — then the next. We lift only the swarm this lesson. Collisions, HUD, and the object pool stay put on purpose; they're noted as future arcs at the bottom.

Step 2 — the Swarm class

A responsibility worth extracting earns its own files. The build already globs src/*.cpp, so new files need no Makefile change.

// NEW FILE: include/swarm.h
#pragma once
#include "game.h"            // for Enemy, MAX_ENEMIES

struct Swarm {
  std::array<Enemy, MAX_ENEMIES> enemies{};
  Vector2 direction{};
  float   speed{};
  int     activeCount{};

  void reset();                              // was InitEnemies
  bool update(float dt);                      // was UpdateEnemies — returns "breached bottom?"
  void draw(const Texture2D& texture) const; // was DrawEnemies
};

Why update returns a bool instead of touching the player

The old UpdateEnemies reached across and wrote state.player.health = 0 when an enemy passed the bottom. That's the swarm reaching into the player's business — a fresh SRP violation inside the very code we're cleaning. Instead the swarm reports a fact ("I breached the bottom edge") and the scene decides the consequence. Same condition-up / decision-at-the-caller pattern you used in lesson 8. The Swarm now knows nothing about Player.

Step 3 — move the bodies in (logic unchanged, just relocated)

// NEW FILE: src/swarm.cpp
#include "swarm.h"
#include "raylib.h"
#include "raymath.h"

// the same constants the swarm needs (move these out of scene_gameplay.cpp)
constexpr int   MAX_ENEMIES_PER_ROW = 11;
constexpr int   COL_PADDING = 80;
constexpr int   ROW_PADDING = 60;
constexpr float ENEMY_SPEED = 50.0f;
constexpr float ENEMY_RADIUS = 15.0f;
constexpr int   ENEMY_SCORE_VALUE = 100;
constexpr float ENEMY_VERTICAL_MOVEMENT = ROW_PADDING / 2.0f;

void Swarm::reset() {
  activeCount = 0;
  direction = {1.0f, 0.0f};
  speed = ENEMY_SPEED;
  Vector2 start = { ((GetScreenWidth() - (MAX_ENEMIES_PER_ROW * COL_PADDING)) / 2.0f)
                    + (COL_PADDING / 2.0f), 100 };
  for (int i = 0; i < MAX_ENEMIES; i++) {
    int col = i % MAX_ENEMIES_PER_ROW, row = i / MAX_ENEMIES_PER_ROW;
    enemies[i] = { .pos = { start.x + col * COL_PADDING, start.y + row * ROW_PADDING },
                   .radius = ENEMY_RADIUS, .scoreValue = ENEMY_SCORE_VALUE, .active = true };
    activeCount++;
  }
}

bool Swarm::update(float dt) {
  bool needDown = false, breached = false;
  int rightEdge = GetScreenWidth() - ENEMY_RADIUS, leftEdge = ENEMY_RADIUS;
  for (Enemy& e : enemies) {
    if (!e.active) continue;
    bool hitR = (e.pos.x + e.radius) >= rightEdge && direction.x > 0;
    bool hitL = (e.pos.x - e.radius) <= leftEdge  && direction.x < 0;
    if (hitR || hitL) { direction.x *= -1.0f; needDown = true; break; }
  }
  if (needDown)
    for (Enemy& e : enemies) {
      e.moveVertically(ENEMY_VERTICAL_MOVEMENT);
      if (e.pos.y > GetScreenHeight()) { breached = true; break; }   // report, don't act
    }
  for (Enemy& e : enemies)
    if (e.active) e.moveHorizontally(direction, speed, dt);
  return breached;
}

void Swarm::draw(const Texture2D& texture) const {
  for (const Enemy& e : enemies) e.draw(texture);
}

Step 4 — shrink GameState

// game.h — after
#include "swarm.h"   // or forward-declare; Swarm is a value member so full type is needed
...
typedef struct GameState {
  Player player;
  std::array<Projectile, MAX_PROJECTILES> bullets;
  Swarm swarm;          // replaces the four loose fields
  bool victory;
  GameResources resources;
  int score;
} GameState;
Include cycle warning: swarm.h includes game.h (for Enemy), and game.h now needs Swarm complete (it's a by-value member). Two files including each other. Fix the ordering: define Enemy before Swarm, and since both live in headers guarded by #pragma once, put the Swarm struct directly in game.h below Enemy rather than a separate swarm.h — OR keep swarm.h but have it #include "game.h" and have game.h not include swarm.h (instead, game.h defines Enemy, then swarm.h defines Swarm, and GameState moves into swarm.h after both). Simplest that compiles: put Swarm in game.h right after Enemy, keep the bodies in src/swarm.cpp. If you hit "incomplete type Swarm", this ordering is why.

Step 5 — rewire scene_gameplay.cpp

Delete InitEnemies, UpdateEnemies, DrawEnemies and their constants (they live in swarm.cpp now). Then:

// InitGameplay
void InitGameplay(GameState* state) {
  state->score = 0;
  state->victory = false;
  state->swarm.reset();          // was: state->activeEnemies = 0; InitEnemies(state);
  InitPlayer(state);
  InitBullets(state);
}
// collisions now reach through the swarm — they stay FREE functions (see box)
void CheckBulletEnemyCollisions(GameState& state) {
  for (Projectile& bullet : state.bullets) {
    if (!bullet.active) continue;
    for (Enemy& enemy : state.swarm.enemies) {
      if (!enemy.active) continue;
      if (CheckCollisionCircles(bullet.pos, bullet.radius, enemy.pos, enemy.radius)) {
        PlaySound(state.resources.explosionSound);
        bullet.active = false; enemy.active = false;
        state.swarm.activeCount--;
        state.score += enemy.scoreValue;
      }
    }
  }
}
// CheckPlayerEnemyCollisions: same — loop state.swarm.enemies, set player.health = 0 on hit
// SceneGameplay::update — the scene owns the consequences
std::unique_ptr<Scene> SceneGameplay::update(GameState& state, float dt) {
  if (state.swarm.update(dt)) state.player.health = 0;   // breach → death decided HERE
  state.player.update(dt);
  PlayerShoot(state);
  UpdateProjectiles(state, dt);
  CheckBulletEnemyCollisions(state);
  CheckPlayerEnemyCollisions(state);

  if (state.swarm.activeCount <= 0) state.victory = true;
  if (state.victory || state.player.health <= 0)
    return std::make_unique<SceneGameover>();
  return nullptr;
}

Why collisions stay free functions (C.4 / C.5, callback to lesson 5)

A bullet-vs-enemy check belongs to neither the Swarm nor the bullet pool — it straddles both. Forcing it into one class would make that class reach into the other's data, recreating the coupling we're removing. The Core Guidelines are explicit: a function that needs the representation of two types should be a non-member. SRP isn't "make everything a method" — it's "each thing has one job." A free collision function whose one job is "mediate between bullets and the swarm" is correct. [C.5: place helper functions in the same namespace as the type they support]

Files you'll touch (checklist)

  1. include/game.h — add Swarm struct (after Enemy); replace the four loose fields in GameState with Swarm swarm;.
  2. src/swarm.cpp (new) — move reset/update/draw bodies + the enemy constants here.
  3. src/scene_gameplay.cpp — delete the three *Enemies functions + enemy constants; rewire InitGameplay, both collision fns (state.swarm.enemies, activeCount), update (breach + victory), and draw.
  4. No Makefile change — src/*.cpp is globbed.
  5. Build & play full cycle: shoot enemies (score++, activeCount→victory), let swarm reach bottom (death), player-enemy touch (death), restart.
Traps to expect: (1) "incomplete type Swarm" → header ordering, see the Step-4 box. (2) Leftover references to state.enemies/state.activeEnemies anywhere → grep for them; all become state.swarm.*. (3) Forgetting swarm.reset() in InitGameplay → restart spawns an empty wave and you instantly "win".
