Lesson 05 · Refactor

Give the bullet its own behaviour

The big structural shift — but one entity at a time. Projectile stops being a passive bag of data and starts owning what it does.

The shift in one sentence

Right now data lives in struct Projectile and behaviour lives in free functions (UpdateProjectiles, the draw loop) that reach into that data. OOP co-locates them: the data and the code that acts on it live in the same class. The bullet knows how to move itself and draw itself.

Why this serves the mission

"Better-structured" means behaviour lives next to the data it touches, so adding a new bullet type or tweaking movement is a local change in one class — not a hunt through free functions that all take GameState*. This is the structural payoff everything so far was setting up: RAII gave you safe ownership, const-correctness gave you draw() const, range-for cleaned the loops that will now call methods.

From struct + functions to a class

Today, in game.h:

struct Projectile {
  Vector2 pos; Vector2 dir; float speed; float radius; bool active;
};

And the behaviour, scattered, in scene_gameplay.cpp:

void UpdateProjectiles(GameState* state, float dt) {
  for (Projectile& b : state->bullets) {
    if (!b.active) continue;
    b.pos = Vector2Add(b.pos, Vector2Scale(b.dir, b.speed * dt));
    b.active = b.pos.y > 0;
  }
}

The move: put the per-bullet behaviour inside the class as member functions. The loop stays outside (it's about the collection); the work for one bullet moves in.

// game.h
struct Projectile {
  Vector2 pos{};
  Vector2 dir{};
  float speed{};
  float radius{};
  bool active{};

  void update(float dt);                 // moves one bullet, deactivates off-screen
  void draw(const Texture2D& tex) const; // draws one bullet; const = won't mutate it
};
Why still struct, not class? In C++ they're nearly identical — the only difference is default access (struct = public, class = private). Since these are simple data-forward entities with public fields, keeping struct is idiomatic and fine. Don't let anyone tell you methods require the class keyword — they don't.

The definitions, in scene_gameplay.cpp

// note the Projectile:: prefix — same idea as GameResources:: in lesson 1
void Projectile::update(float dt) {
  if (!active) return;
  pos = Vector2Add(pos, Vector2Scale(dir, speed * dt));
  active = pos.y > 0;
}

void Projectile::draw(const Texture2D& tex) const {
  if (!active) return;
  DrawOffset(tex, pos, WHITE);
  DrawCircleLinesV(pos, radius, RED);
}

Inside a member function, pos/active/etc. refer to this object's fields — no state->, no parameter. That implicit "this object" is the whole point: the method already knows which bullet it's working on.

The loops shrink to calls

void UpdateProjectiles(GameState* state, float dt) {
  for (Projectile& b : state->bullets) b.update(dt);
}
void DrawProjectiles(const GameState* state) {
  for (const Projectile& b : state->bullets)
    b.draw(state->resources.laserTexture);
}

The const& in the draw loop now requires draw() to be a const method — a non-const method can't be called on a const reference. That's lesson 4 paying off: draw() const and const Projectile& fit together like a key in a lock.

  4. Build & play. Bullets fire, move, vanish off-screen exactly as before.
