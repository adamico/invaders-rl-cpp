Lesson 04 · Refactor

Say what won't change

A small, safe step before classes: mark the read-only code const so the compiler enforces the difference between "looks at" and "changes."

The idea

const is a promise to the compiler — and to the next reader — that a thing won't be modified. Used well, it turns a whole category of bugs into compile errors: if you accidentally write to something you meant only to read, the build fails instead of the game misbehaving. It's also documentation that can't go stale: a function signature with const tells you, at a glance, whether it mutates. [Core Guidelines: Con — constants and immutability]

Why now, why this matters for classes

Next lesson your draw functions become methods: enemy.draw(). A draw method must be a const method — void draw() const — meaning "calling this can't change the enemy." You can't write correct const methods if you haven't internalised which operations are read-only. This lesson builds that instinct on the code you already have.

Two places to apply it

1 — Read-only range-for loops

Your draw and scan loops only read each element. Say so:

// DrawEnemies — reads pos/active, never writes
- for (Enemy& enemy : state->enemies)
+ for (const Enemy& enemy : state->enemies)

Still a reference (no copy, no overhead) — but now const. If a future edit tries enemy.active = false inside that loop, the compiler stops you. The loops that do mutate (movement, collisions) keep plain Enemy&. The const is what distinguishes them at a glance.

2 — Pointers that only read state

Several functions take GameState* state but never modify it — they only inspect. DrawEnemies, DrawProjectiles, DrawPlayer, DrawGameplay are pure readers. Mark the pointee const:

- void DrawEnemies(GameState* state)
+ void DrawEnemies(const GameState* state)

Read it right-to-left: "state is a pointer to a const GameState" — you may follow the pointer and read, but not write through it. Try state->score = 0 inside and the build fails. (Update the matching declarations in scene_gameplay.h too.)

The distinction to feel: const GameState* state = can't change what it points to (the game). GameState* const state = can't repoint it, but can change the game. You want the first for read-only functions. The const goes left of the *.

Your task feedback loop

  1. Add const to the loop variable in every read-only range-for: the draw loops, and any scan that doesn't write. Leave mutating loops as plain &.
  2. Change the four Draw* functions to take const GameState* — both the definition and the declaration in the header.
  3. Build. Let the compiler teach you: if a function you marked const actually writes somewhere, you'll get an error pointing at the exact line. That's a real finding — either the const is wrong, or that write shouldn't be there.
  4. Play the game. Behaviour is identical — const changes nothing at runtime. The win is entirely at compile time.
If the build fails inside a Draw* function: read what it's writing. DrawText(TextFormat(...)) only reads — fine. But if you find a draw function sneakily mutating state, you've just discovered a design smell the const exposed. Tell me what you found.
