Lesson 16 · Refactor · Encapsulation capstone

Make items private — earn it

You asked the right question: can Pool::items finally go private? Not yet — five call sites still reach in. The fix isn't to hide the data and break them; it's to give Pool a real container interface so nobody needs the raw array. Then privacy is free.

Why bother hiding a field?

items being public means any code, anywhere, can do anything to it — reassign the whole array, resize assumptions, leave it inconsistent. Privacy buys two things: (1) you control the surface — callers use the operations you sanctioned, not arbitrary field pokes; (2) you keep the freedom to change the representation later (a free-list, a bitset of active flags, a different array size) without touching a single caller, because they never depended on the representation — only the interface. "Expose operations, not data" is the whole game. [Core Guidelines C.9: minimise exposure of members; C.131: avoid trivial getters/setters by exposing meaningful operations]

So two new pieces of interface — range-for support and operator[] — cover everything. Add them, and the raw items has no remaining users outside the class.

Tool 1: make Pool iterable (the range-for protocol)

A range-based for (x : container) is just sugar. The compiler rewrites it to call container.begin() and container.end() and walk between them. So "being iterable" means: provide begin() and end(). Your underlying std::array already has them — just forward:

// pool.h
  auto begin()       { return items.begin(); }
  auto end()         { return items.end(); }
  auto begin() const { return items.begin(); }   // const overloads → const range-for works too
  auto end()   const { return items.end(); }

Now for (Enemy& e : pool) works — and crucially, it's a real loop in the caller, so break and continue work normally. That's why it serves the edge-detect and breach loops that eachActive couldn't. [cppreference: range-for]

Tool 2: operator[] for indexed access

Swarm::reset writes specific slots by index to build the formation. Overload the subscript operator so pool[i] means pool.items[i]:

// pool.h
  T&       operator[](std::size_t i)       { return items[i]; }
  const T& operator[](std::size_t i) const { return items[i]; }

Operator overloading lets your type use built-in syntax. Return T& so pool[i] = {...} assigns into the slot; the const overload returns const T& for read-only contexts (const-correctness again, lesson 4). [cppreference: operator overloading]

Now hide the field — and switch struct to class

// pool.h — final shape
template <typename T, std::size_t N>
class Pool {
public:
  T* acquire();                                  // (bodies as before)
  int activeCount() const;
  template <typename F> void eachActive(F fn);
  template <typename F> void eachActive(F fn) const;
  auto begin();  auto end();                     // + const overloads
  auto begin() const;  auto end() const;
  T& operator[](std::size_t i);
  const T& operator[](std::size_t i) const;

private:
  std::array<T, N> items{};                       // nobody outside Pool can touch this now
};

struct vs class — the only real difference

They are identical except the default access level: struct members are public by default, class members private. The convention (C.2, C.8): use struct for a passive bundle of public data with no invariant (your Enemy, Projectile — plain data); use class once you have private state behind a deliberate interface. Pool just graduated. Your entity types stay struct — they really are just public fields, and that's fine.

Convert the five sites

// swarm.cpp — reset(): index write, and the all-touch is implicit in the index loop
pool[enemyIndex] = { .pos = {...}, .radius = ENEMY_RADIUS, ... };   // was pool.items[enemyIndex]

// swarm.cpp — edge-detect & breach loops (these break; range-for is fine)
for (Enemy& enemy : pool) { ... break; ... }                // was : pool.items

// projectile_pool.cpp — reset() clears all slots
for (auto& projectile : pool) projectile = { .active = false };

// scene_gameplay.cpp — collisions
for (Projectile& bullet : state.projectilePool.pool) { ... }     // was .pool.items
for (Enemy& enemy : state.swarm.pool) { ... }                   // ×2

Build. If anything still names .items outside pool.h, the compiler now refuses it — that error is the proof the encapsulation is real.

"Isn't exposing begin/end/[] just as leaky as public items?"

Fair challenge — and no. Public items hands out the array object: callers can swap it wholesale, and they're welded to "it's a std::array<T,N>." The interface hands out access patterns (walk the elements, index one) without naming the container. Swap the storage tomorrow and as long as begin/end/[] still work, every caller compiles unchanged. You're exposing what you can do, not what it is. That's the line between interface and representation.

Files you'll touch (checklist)

  1. include/pool.h — add begin/end (×2 const) and operator[] (×2 const); switch struct→class with public:/private:; move items under private:.
  2. src/swarm.cpp — pool.items[i]→pool[i]; the two break-loops : pool.items→: pool.
  3. src/projectile_pool.cpp — reset()'s loop : pool.items→: pool.
  4. src/scene_gameplay.cpp — the three collision loops .pool.items→.pool.
  5. Build & play full cycle — behaviour identical; items now unreachable from outside.
Traps: (1) Provide the const overloads of begin/end or any const range-for over a pool won't compile. (2) operator[] returning T& (not by value) — you must assign into the slot. (3) acquire/eachActive use items internally — that's fine, they're members; only outside access is blocked. (4) If you kept Pool a struct, you must still write private: before items — the access label is what matters, not the keyword.
