Lesson 15 · Refactor · Lambdas

Hand the loop a function

The same four lines keep recurring: for (auto& x : pool.items) if (!x.active) continue; …. Give Pool one method that owns the loop and the active-check, and pass it the body as a lambda. New mechanic: lambdas — anonymous functions you write inline.

The repetition, one more time

// swarm.cpp — move active enemies
for (Enemy& e : pool.items) { if (!e.active) continue; e.moveHorizontally(direction, speed, dt); }
// projectile_pool.cpp — update active bullets
for (auto& p : pool.items)  { if (!p.active) continue; p.update(dt); }   // (Projectile::update self-guards too)
// draw, in both
for (const auto& x : pool.items) { if (!x.active) continue; x.draw(texture); }

The loop + the if (!active) filter is identical every time; only the body differs. That "everything's the same except this one block" shape is exactly what a lambda lets you factor out: keep the loop in one place, pass the varying block in.

The tool: a member that takes a callable

// pool.h — two overloads (mutating + read-only)
template <typename T, std::size_t N>
struct Pool {
  std::array<T, N> items{};
  ...
  template <typename F> void each_active(F fn) {
    for (T& item : items) if (item.active) fn(item);
  }
  template <typename F> void each_active(F fn) const {   // for const methods like draw
    for (const T& item : items) if (item.active) fn(item);
  }
};

F is a template parameter — each_active accepts any callable: a function pointer, a functor, or (what you'll pass) a lambda. Because F is a template, the call fn(item) is resolved and inlined at compile time — zero overhead, unlike std::function which would box it behind an indirection. Same compile-time-generic idea as your overlaps and Pool itself. [Core Guidelines T.40: use function objects to pass operations to algorithms]

The new syntax: a lambda

[&](Enemy& e) { e.moveHorizontally(direction, speed, dt); }
 │     │            │
 │     │            └─ body — the per-element work
 │     └─ parameter list — what each_active hands you
 └─ capture list — which surrounding variables the body may use

A lambda is an anonymous function you define right where you use it. The one genuinely new part is the capture list [&]: the body needs direction, speed, dt — locals from the enclosing function. [&] means "capture them by reference," so the lambda reads/writes the real variables. ([=] would copy them; [] captures nothing and the body couldn't see them at all.) For these per-frame loops you want [&]: no copies, and you're operating on live state. [cppreference: lambda expressions]

Apply it — the do-to-all loops

// swarm.cpp — the horizontal-move loop
pool.each_active([&](Enemy& e) { e.moveHorizontally(direction, speed, deltaTime); });

// swarm.cpp — draw (const overload kicks in because draw() is const)
pool.each_active([&](const Enemy& e) { e.draw(texture); });

// projectile_pool.cpp — update and draw
pool.each_active([&](Projectile& p) { p.update(deltaTime); });
pool.each_active([&](const Projectile& p) { p.draw(texture); });

The loop and the active-filter now exist once, inside each_active. Each call site states just its intent: "for each active enemy, move it." The if (!active) continue; boilerplate is gone from these four sites.

Where each_active does NOT fit — and why that's the lesson

Two loops in Swarm::update break early: the edge-detection loop (flip direction on the first enemy at the boundary) and the breach loop (stop at the first enemy past the bottom). A lambda cannot break the caller's loop — it's a separate function; all it can do is return from itself. So each_active is the wrong tool there; those stay as raw for loops. The right tool for "stop at the first match" is a find/any-shaped helper (returns a pointer or bool), not a for-each. Knowing which loops are "do to all" (→ each_active) versus "find one and stop" (→ leave it, or a different helper) is the actual skill — don't force every loop through the same abstraction.

Mutating vs const: why two overloads

draw() is a const method, so when it runs the Pool is const and so are its items — the lambda must take const Enemy&. The non-const each_active wouldn't compile there. The two overloads let the compiler pick automatically based on whether the call site is in a const context. This is const-correctness (lesson 4) flowing through a template.

Files you'll touch (checklist)

  1. include/pool.h — add the two each_active overloads (mutating + const).
  2. src/swarm.cpp — convert the horizontal-move loop and the draw loop to pool.each_active(...). Leave the edge-detection and breach loops as raw for (they break).
  3. src/projectile_pool.cpp — convert the update and draw loops.
  4. No Makefile change; pool.h stays header-only.
  5. Build & play full cycle: movement, firing, drawing, victory, breach all identical.
Traps: (1) draw's lambda must take const Enemy& / const Projectile&, or the const overload won't match. (2) Don't convert the breaking loops — a lambda can't break; if you find yourself wanting to, that loop isn't an each_active. (3) Capture [&], not [=] — you're mutating live state and don't want copies. (4) reset() in both classes touches inactive slots too (it builds/clears all of them), so it stays a plain loop over pool.items — each_active would skip the inactive ones.
