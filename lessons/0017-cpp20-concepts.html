Lesson 17 · Refactor · C++20 concepts

Name the contract your templates already assume

Your overlaps<A,B> accepts any two types and hopes they have .pos and .radius. Pool<T,N> accepts any T and hopes it has .active. Today those hopes become named, checked requirements — so a wrong type is rejected at the call boundary with a readable message, not deep inside the body.

The problem: invisible, unchecked requirements

Since L13 your templates do compile-time duck typing: "if it has .pos and .radius, it works." That's powerful but the contract is implicit — written nowhere, enforced only by the body happening to compile. Two costs:

  1. Bad diagnostics. Pass a type missing .radius and the error points inside overlaps at CheckCollisionCircles — "no member named radius" — blaming the implementation, not your call. In real codebases these errors are pages long.
  2. No documentation. Reading template <typename A, typename B> tells you nothing about what A must provide. The requirement lives only in the author's head.

A concept is a named, compile-time predicate over a type that fixes both: it documents the requirement and moves the check to the boundary. [cppreference — Constraints and concepts]

Anatomy of a concept

// a concept is a compile-time bool predicate over a type T
template <typename T>
concept Collidable = requires(const T t) {
  { t.pos }    -> std::convertible_to<Vector2>;
  { t.radius } -> std::convertible_to<float>;
};

Reading it piece by piece:

PieceMeaning
concept Name =declares a named predicate; the right-hand side is a constexpr bool over the type.
requires(const T t)a requires-expression: introduces a hypothetical t so you can write expressions that must be valid. Nothing is actually constructed.
{ t.pos };a simple requirement: this expression must compile.
{ t.pos } -> std::convertible_to<Vector2>;a compound requirement: expression must compile and its result type must satisfy the trailing constraint.

std::convertible_to is a standard library concept from <concepts>. The whole thing evaluates to true only for types that have both members with compatible types.

Using a concept to constrain a template

Swap the bare typename for the concept name. Three equivalent spellings — pick the terse one:

// collision.h
#pragma once
#include <concepts>
#include "raylib.h"

template <typename T>
concept Collidable = requires(const T t) {
  { t.pos }    -> std::convertible_to<Vector2>;
  { t.radius } -> std::convertible_to<float>;
};

// constrained: A and B must both be Collidable
template <Collidable A, Collidable B>
bool overlaps(const A& a, const B& b) {
  return CheckCollisionCircles(a.pos, a.radius, b.pos, b.radius);
}

template <Collidable A> reads "A is some type satisfying Collidable" — exactly like template <typename A> but with the constraint attached. (Equivalent: a trailing requires Collidable<A> && Collidable<B> clause, or the abbreviated bool overlaps(const Collidable auto& a, const Collidable auto& b). Same effect; the two-param form above is clearest here.) [Core Guidelines T.10/T.11: specify concepts for template arguments]

Before / after the error message

Suppose someone calls overlaps(score, enemy) where score is an int.

Without concept (now)With concept
Error deep inside: request for member 'pos' in 'a', which is of non-class type 'const int' — points at collision.h's body.Error at the call: constraints not satisfied … the required expression 't.pos' is invalid — names Collidable and your call site.

The constraint turns "the body failed to compile" into "you passed the wrong kind of thing." That's the entire payoff: same machine code, much better failure mode, self-documenting signature.

Second concept: Poolable for Pool<T,N>

Pool assumes T has a bool active — acquire tests !item.active, activeCount/eachActive test item.active. Name it:

// pool.h — add near the top
#include <concepts>

template <typename T>
concept Poolable = requires(T t) {
  { t.active } -> std::convertible_to<bool>;
};

template <Poolable T, std::size_t N> class Pool {
  // ...body unchanged...
};

Now Pool<Enemy, 55> compiles (Enemy has active); Pool<Vector2, 4> is rejected at the declaration with "Vector2 does not satisfy Poolable" instead of an error inside acquire. The non-type param std::size_t N stays exactly as it was — concepts constrain type params; N is a value.

Why requires(T t) here but requires(const T t) for Collidable? No deep reason in this code — both members are just read. Use const T when every requirement is read-only (documents that you don't mutate to test the contract). If a requirement needed a non-const operation (e.g. { t.reset() }), you'd drop the const. Prefer the most-const form that still compiles.

The one build change: c++17 → c++20

Concepts are a C++20 feature. Your Makefile line 19 is the only thing standing in the way:

# Makefile, line 19 — change c++17 to c++20
CXXFLAGS = -std=c++20 -Wall -I$(RAYLIB_SRC) -Iinclude

Nothing else in the build moves. Everything you wrote in L1–L16 is valid C++20 unchanged — the standard is (almost entirely) a superset. If <concepts> or std::convertible_to isn't found after the bump, your compiler is too old; check c++ --version (need Clang 10+/GCC 10+, both fine on current macOS).

Files you'll touch (checklist)

  1. Makefile — line 19: -std=c++17 → -std=c++20.
  2. include/collision.h — #include <concepts>; add the Collidable concept; change template <typename A, typename B> → template <Collidable A, Collidable B>.
  3. include/pool.h — #include <concepts>; add the Poolable concept; change template <typename T, std::size_t N> → template <Poolable T, std::size_t N>.
  4. Build & play full cycle — behaviour identical (concepts are compile-time only; zero runtime cost, zero codegen change).
  5. Prove it (optional): temporarily add overlaps(state.score, state.player); somewhere, build, read the new error, delete it. Seeing the boundary error is the lesson.
Traps: (1) Forgetting #include <concepts> — std::convertible_to won't resolve. (2) Concepts constrain type params only; don't try to wrap std::size_t N. (3) The -> in a compound requirement is not a return-type arrow — it's "result type must satisfy this constraint"; the thing after it must itself be a concept, not a plain type (use std::convertible_to<Vector2>, not bare Vector2). (4) This is polish: it changes diagnostics and documentation, not behaviour — don't expect the game to play differently.
