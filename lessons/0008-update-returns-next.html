Lesson 08 · Refactor · Kill the last switch

Let a scene name its own successor

The factory switch in makeScene and the whole GameScene enum exist for one reason: to turn a label back into a scene object. What if update just handed back the next scene directly?

The indirection you're paying for

Today a transition is a three-step dance:

1. a scene writes   state.currentScene = GameScene::GAMEPLAY;   // label
2. SceneManager sees the label changed
3. makeScene(label)  switch → make_unique<SceneGameplay>()        // label → object

The enum is a stand-in for "which scene object to build." But the scene already knows exactly where it wants to go — it just isn't allowed to say so directly. Remove the middleman: let update return the next scene, or nothing if it wants to stay.

The tool: return a unique_ptr, use nullptr as "stay"

// scene.h
struct Scene {
  virtual std::unique_ptr<Scene> update(GameState& state, float dt) = 0;
  virtual void draw(const GameState& state) const = 0;
  virtual ~Scene() = default;
};

The return value carries the decision: a non-null pointer means "replace me with this"; nullptr means "no transition, keep running me." Returning a unique_ptr by value moves ownership out of the function — the same move you already trust from makeScene, now done per-scene. [Core Guidelines F.20: prefer returning a value to writing through an out-parameter]

// scene_title.cpp — was: state.currentScene = GameScene::GAMEPLAY;
std::unique_ptr<Scene> SceneTitle::update(GameState& state, float) {
  if (IsKeyPressed(KEY_SPACE)) return std::make_unique<SceneGameplay>();
  return nullptr;   // stay on the title
}

SceneManager::tick collapses — no more shown_ mirror, no transition compare:

void SceneManager::tick(GameState& state, float dt) {
  if (auto next = current_->update(state, dt))   // truthy = a scene was returned
    current_ = std::move(next);                 // move-assign: old deleted, new installed
  current_->draw(state);
}

The class drops to a single member, current_. shown_ is gone because there's nothing to keep in sync anymore — the transition is the return value, atomic by construction.

Trap: the if line is the update call — don't call it twice

if (auto next = current_->update(state, dt)) already runs update once this frame and captures what it returned. A leftover second current_->update(state, dt) before draw double-steps your simulation — every entity moves twice per frame, so the player appears to move at double speed. The body is exactly three lines: the if, the move inside it, and draw. Nothing else.

Transitions triggered deep in the call tree

Here's the part the obvious examples skip. Your gameplay scene doesn't decide "game over" in update directly — it's detected inside helpers like UpdateEnemies and CheckPlayerEnemyCollisions, which return void. A void helper buried two calls deep cannot return a scene up to update. So don't try.

Split the two jobs: helpers mutate condition state; only update returns a scene. A helper that used to write state.currentScene = GAMEOVER instead sets the underlying condition (e.g. state.player.health = 0, or a state.victory flag). Then update — the one function that owns the return type — reads those conditions after running the frame and produces the successor at a single decision point:

std::unique_ptr<Scene> SceneGameplay::update(GameState& state, float dt) {
  state.player.update(dt);
  UpdateEnemies(state, dt);              // may set health = 0 (death) internally
  PlayerShoot(state);
  UpdateProjectiles(state, dt);
  CheckBulletEnemyCollisions(state);
  CheckPlayerEnemyCollisions(state);     // may set health = 0 internally

  if (state.activeEnemies <= 0) state.victory = true;       // win condition
  if (state.victory || state.player.health <= 0)            // ONE decision point
    return std::make_unique<SceneGameover>();
  return nullptr;                                          // otherwise: stay
}
Why not a helper that returns the scene? You could make CheckPlayerEnemyCollisions return unique_ptr<Scene> too and bubble it up — but then every helper's signature changes and update becomes a chain of "if a helper returned something, return that." Conditions-in-state keeps the helpers simple and the transition logic in one readable place. And beware the failure that compiles but misbehaves: a helper like std::unique_ptr<Scene> check(...) that only returns inside an if and falls off the end on the other path is undefined behaviour — every control path must return.

What gets deleted

  • makeScene and its switch — gone entirely. That empties src/scene.cpp: delete the file (the build globs src/*.cpp, so no Makefile edit needed). scene.h stays — it still declares the Scene base.
  • The GameScene enum — gone from game.h (nothing references a scene by label now).
  • state.currentScene the field — gone from GameState in game.h.
  • SceneManager::shown_ — gone. The header collapses to one member:
// scene_manager.h — final shape
class SceneManager {
public:
  explicit SceneManager(std::unique_ptr<Scene> first);
  void tick(GameState& state, float dt);
private:
  std::unique_ptr<Scene> current_;   // shown_ deleted
};
// scene_manager.cpp — ctor takes the object, no label, no mirror
SceneManager::SceneManager(std::unique_ptr<Scene> first)
  : current_(std::move(first)) {}

The ctor now takes the first scene as an object, not a label:

// main.cpp
SceneManager scenes(std::make_unique<SceneTitle>());
while (!WindowShouldClose()) scenes.tick(state, GetFrameTime());

The honest trade-off — read before you decide you like this

Lesson 6 centralised "what scenes exist" into one factory switch — one place to look. This lesson distributes that knowledge: SceneTitle now #includes and names SceneGameplay; SceneGameover names SceneTitle. Each scene is coupled to its successors. Is that better? It depends what you value: the factory had low coupling, central knowledge; this has high cohesion (a scene fully owns its own transitions) at the cost of scene-to-scene coupling. For a small linear flow it's a clear win — the title genuinely does know it leads to gameplay. For a large graph of scenes that all jump to each other, the central factory may age better. Neither is "correct"; recognising the axis is the skill.

The include trap this creates

SceneTitle's header doesn't need SceneGameplay — but its .cpp does, because make_unique<SceneGameplay>() needs the complete type to construct it. Keep the #include "scene_gameplay.h" in scene_title.cpp, not the header, to avoid include cycles (Title→Gameplay→… ) and keep headers light. Returning unique_ptr<Scene> only needs Scene to be complete, which scene.h already gives you. [SF.10 / general: include what you use, in the .cpp where possible]

Why nullptr and not std::optional?

A unique_ptr is already nullable — it has a built-in empty state that means exactly "no object." Wrapping it in std::optional<unique_ptr<Scene>> would give you two ways to say "nothing" (empty optional vs null pointer) — redundant and confusing. Use optional when a type has no natural empty value (an int, a Vector2). A pointer isn't one of those. [cppreference: optional]

Your task feedback loop

  1. Change Scene::update's return type to std::unique_ptr<Scene> in scene.h and every override.
  2. In each scene's update: return make_unique<NextScene>() where it used to set the enum; return nullptr otherwise. Add the successor's #include to the .cpp.
  3. Rewrite SceneManager::tick to the if (auto next = …) current_ = std::move(next); form. Delete shown_ and change the ctor to take std::unique_ptr<Scene>.
  4. Delete makeScene → empties src/scene.cpp, so rm src/scene.cpp. Delete the GameScene enum and the state.currentScene field from game.h. Also update each scene header's update return type — not just the .cpp (the headers declare it too).
  5. In main: SceneManager scenes(std::make_unique<SceneTitle>());.
  6. Build & play the full cycle. Watch the restart path — see the trap.
Restart trap: SceneGameover::update used to do two things — reset state (InitGameplay) and switch to title. Keep both: call InitGameplay(&state) then return std::make_unique<SceneTitle>();. If you only return the scene and drop the reset, restart shows a title over a dead game state. Lifetime trap: after current_ = std::move(next) the old scene (i.e. this inside the running update) is deleted — so don't touch any member of the old scene after returning the new one. You already return last, so you're fine; just know why.
