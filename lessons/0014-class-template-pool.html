Lesson 14 · Refactor · Class templates

One pool, any element

Last lesson templated a function. Now template a class. Swarm and ProjectilePool both wrap a fixed array of flag-activated things and both hunt for free slots by hand. Extract that shared machinery into Pool<T, N> — and watch a stored invariant disappear entirely.

What the two share — and what they don't

Be precise before extracting. Swarm and ProjectilePool have in common only: a std::array<T, N>, "find the first inactive slot," and "how many are active." Their behaviour — the swarm's edge-bounce/descend, the pool's fire — is bespoke and stays put. So this isn't inheritance; it's composition: each class will contain a Pool and keep its own logic on top. [Core Guidelines C.129/ C.8: prefer composition; templates for shared mechanism]

The class template

// NEW FILE: include/pool.h
#pragma once
#include <array>
#include <cstddef>

template <typename T, std::size_t N>
struct Pool {
  std::array<T, N> items{};

  T* acquire() {                 // first inactive slot, or nullptr if full
    for (T& item : items)
      if (!item.active) return &item;
    return nullptr;
  }

  int active_count() const {     // DERIVED — never stored, so it can never drift
    int n = 0;
    for (const T& item : items)
      if (item.active) ++n;
    return n;
  }
};

Two template parameters of different kinds: typename T (a type — what's pooled) and std::size_t N (a non-type template parameter — an actual compile-time number, the array size). Pool<Enemy, 55> and Pool<Projectile, 100> are two distinct classes the compiler generates from one definition. Like last lesson, it's duck-typed: any T with an active member works; nothing else is required. And like last lesson — the whole template lives in the header, no pool.cpp.

The payoff: a derived count beats a maintained one (sequel to lesson 12)

Lesson 12 taught you to maintain activeCount safely by funnelling all changes through Swarm::deactivate — keeping a stored value and its invariant in sync. The deeper move is to ask: why store it at all? active_count() computes it on demand by counting active flags. There is no stored field, so there is no invariant to keep — it is impossible for the count to be wrong. (Counting 55 flags per frame is free here.) The safest state is state you don't duplicate. Swarm can now delete its int activeCount field entirely.

Swarm: compose the pool, drop the stored count

// swarm.h — before → after
struct Swarm {
  std::array<Enemy, MAX_ENEMIES> enemies{};
  int activeCount{};
  Pool<Enemy, MAX_ENEMIES> pool;
  Vector2 direction{};
  int speed{};

  void reset();
  bool update(float deltaTime);
  void draw(const Texture2D& texture) const;
  void deactivate(Enemy& enemy);
  int activeCount() const { return pool.active_count(); }   // now a method, not a field
};

Add #include "pool.h" to swarm.h. Then in swarm.cpp, every bare enemies becomes pool.items, and the count bookkeeping vanishes:

// swarm.cpp — reset(): drop the two activeCount lines, loop pool.items
void Swarm::reset() {
  direction = {1.0f, 0.0f};
  speed = ENEMY_SPEED;
  Vector2 start = { ... };
  for (int i = 0; i < MAX_ENEMIES; i++) {
    int col = i % MAX_ENEMIES_PER_ROW, row = i / MAX_ENEMIES_PER_ROW;
    pool.items[i] = { .pos = {...}, .radius = ENEMY_RADIUS,
                     .scoreValue = ENEMY_SCORE_VALUE, .active = true };
    // activeCount++  ← gone
  }
}

// update() / draw(): for (Enemy& e : pool.items)  — same loops, new source

void Swarm::deactivate(Enemy& enemy) {
  enemy.active = false;     // no count to maintain now — the guard/-- are gone
}
Tell-Don't-Ask still holds. Keep Swarm::deactivate(Enemy&) as the verb collision code calls — its body just got trivial. The reason it existed (protect the count) is gone, but the interface staying put means the collision call sites don't change at all.

ProjectilePool: compose, and let acquire simplify fire

// projectile_pool.h
struct ProjectilePool {
  std::array<Projectile, MAX_PROJECTILES> projectiles{};
  Pool<Projectile, MAX_PROJECTILES> pool;
  void reset();
  void update(float deltaTime);
  void draw(const Texture2D& texture) const;
  bool fire(Vector2 from, Vector2 dir = {0.0f, -1.0f});
};
// projectile_pool.cpp — fire() collapses onto acquire() + the existing Projectile::spawn
bool ProjectilePool::fire(Vector2 from, Vector2 dir) {
  Projectile* p = pool.acquire();
  if (!p) return false;            // pool full — nothing fired
  p->spawn(from, dir);             // spawn already sets pos/dir/speed/radius/active (lesson 5)
  return true;
}
// reset/update/draw: loop pool.items instead of projectiles

Rewire the call sites (only three reach inside)

// scene_gameplay.cpp
for (Projectile& bullet : state.projectilePool.pool.items) { ... }   // was .projectiles
for (Enemy& enemy : state.swarm.pool.items) { ... }              // was .enemies (×2)
if (state.swarm.activeCount() <= 0) state.victory = true;          // was .activeCount (now a call)

Everything else — reset(), fire(), update(), draw(), deactivate() — is unchanged, because those are the classes' public surface and it didn't move. Good encapsulation is what keeps this refactor small.

Reaching state.swarm.pool.items is two dots deep — note it

That's a mild Law-of-Demeter poke (the collision loop reaches through swarm into its pool's items). It's acceptable for now, but it's the thread the next lesson pulls: a generic each_active(fn) on Pool that takes a callable (a lambda), so callers say swarm.pool.each_active([&](Enemy& e){ ... }) and the bare if (!active) continue; loops disappear for good. That's lambdas — a new mechanic — so it gets its own lesson.

Files you'll touch (checklist)

  1. include/pool.h (new) — the Pool<T, N> template, header-only.
  2. include/swarm.h — #include "pool.h"; swap enemies+activeCount field for Pool<Enemy, MAX_ENEMIES> pool;; add int activeCount() const method.
  3. src/swarm.cpp — enemies → pool.items throughout; delete activeCount = 0 / ++ / --; deactivate becomes one line.
  4. include/projectile_pool.h + src/projectile_pool.cpp — same swap; fire uses pool.acquire() + spawn.
  5. src/scene_gameplay.cpp — the three reach-in sites above (.pool.items ×3, .activeCount() ×1).
  6. No Makefile change. Build & play full cycle: shooting caps at the pool size, victory fires once when the swarm clears, breach + collision still kill.
Traps: (1) activeCount is now a method — the victory line needs (); the compiler will catch a missed one. (2) Don't write pool.cpp — header-only template (linker error otherwise, same as lesson 13). (3) If fire previously set speed/radius inline, make sure Projectile::spawn still sets them so the switch to p->spawn() doesn't drop fields. (4) Grep for any leftover bare enemies / projectiles / activeCount (field form).
