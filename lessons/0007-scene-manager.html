Lesson 07 · Refactor · Beyond the roadmap

A class to own the scenes

Your main loop still does bookkeeping it shouldn't: it holds the unique_ptr, mirrors the current scene in a shown variable, and polls for transitions. That's a responsibility, and it has a name. Give it a class.

What's wrong with main right now

Look at what main currently carries (from lesson 6):

std::unique_ptr<Scene> current = makeScene(GameScene::TITLE);
GameScene shown = GameScene::TITLE;            // a mirror of "which scene is live"

while (!WindowShouldClose()) {
  if (state.currentScene != shown) {           // transition bookkeeping
    current = makeScene(state.currentScene);
    shown = state.currentScene;                // keep the mirror in sync — easy to forget
  }
  current->update(state, GetFrameTime());
  current->draw(state);
}

current and shown are an invariant pair: shown must always name the type current points at. Right now nothing enforces that — if a future edit updates one and forgets the other, the game silently shows the wrong scene. Whenever two variables must move together, that's a class waiting to be born. main's job is to run the game, not to babysit scene transitions. This is the single-responsibility principle, and the same C.4 judgment you used in lesson 5: a function (or here, a loop) belongs with the data whose invariant it protects.

The tool: a coordinator class

Lesson 5 gave you entity classes (Player, Enemy) — small things that own their own data. SceneManager is a different animal: a coordinator. It owns no game data; it owns the relationship between "which scene should run" and "which scene object exists," and exposes one narrow verb to the outside world.

// scene_manager.h
#pragma once
#include "scene.h"
#include <memory>

class SceneManager {
public:
  explicit SceneManager(GameScene start);            // build the first scene
  void tick(GameState& state, float dt);             // one frame: transition? update + draw

private:
  std::unique_ptr<Scene> current_;
  GameScene shown_;                                  // the mirror — now an invariant of the class
};

Why the constructor matters here

The constructor's job is to establish the invariant before anyone can observe the object: build current_ and set shown_ to match, in one place, atomically. After the constructor runs, it is impossible to hold a SceneManager whose mirror is out of sync — because the only code that touches shown_ lives inside the class. That's the whole point of private: it's not secrecy, it's a guarantee. Same idea as GameResources in lesson 1 — the constructor owns the setup so callers can't get it wrong. [Core Guidelines C.40 / C.41: a constructor should create a fully-initialised object]

explicit on a single-argument constructor blocks a surprising implicit conversion (a bare GameScene silently becoming a SceneManager). Habit worth keeping: single-argument constructors are explicit by default. [C.46]

The implementation — move the bookkeeping in, unchanged

// scene_manager.cpp
#include "scene_manager.h"

SceneManager::SceneManager(GameScene start)
  : current_(makeScene(start)), shown_(start) {}   // member init list — invariant set at birth

void SceneManager::tick(GameState& state, float dt) {
  if (state.currentScene != shown_) {              // a scene requested a transition
    current_ = makeScene(state.currentScene);      // move-assign: old scene deleted, new moved in
    shown_ = state.currentScene;
  }
  current_->update(state, dt);                      // still virtual dispatch — unchanged
  current_->draw(state);
}

Notice the logic is identical to what was in main — you've relocated it, not rewritten it. The move semantics from lesson 6 still happen inside tick; they're now an implementation detail nobody outside the class needs to think about. That's the win: encapsulation lets the same code stop being everyone's problem.

main shrinks to what it's actually for

// main.cpp
GameState state = {};
InitGameplay(&state);
state.currentScene = GameScene::TITLE;

SceneManager scenes(GameScene::TITLE);

while (!WindowShouldClose()) {
  scenes.tick(state, GetFrameTime());   // that's the whole loop body
}

The unique_ptr, the shown mirror, the transition if — all gone from main. The loop now reads as one sentence: until the window closes, tick the scenes. You can delete #include <memory> from main.cpp (it no longer touches a smart pointer directly) and add #include "scene_manager.h".

The last switch is still alive — on purpose

makeScene's factory switch hasn't moved; SceneManager calls it. That's fine — this lesson is about ownership, not about killing the switch. Removing it needs a different idea: letting update return the next scene instead of writing state.currentScene. That's lesson 8. One change at a time, game playable after each.

Your task feedback loop

  1. Create include/scene_manager.h with the SceneManager class (ctor + tick, two private members).
  2. Create src/scene_manager.cpp: ctor uses a member initialiser list; tick holds the transition if + the two virtual calls.
  3. Add scene_manager.cpp to your build (Makefile / CMake — wherever the other src/*.cpp are listed).
  4. In main.cpp: replace the unique_ptr + shown + loop body with a SceneManager scenes(...) and a one-line scenes.tick(...).
  5. Delete the now-dead lines from main and fix its includes.
  6. Build & play the full cycle: Title → Gameplay → Gameover → Title. Behaviour must be identical to before.
Traps to expect: (1) forgetting to add scene_manager.cpp to the build → linker error undefined reference to SceneManager::tick (a link error, not a compile error — different beast, worth recognising). (2) Initialising current_ in the constructor body with = instead of the init list still works here, but prefer the init list — it's the idiom and it matters once members lack default constructors. (3) Leaving state.currentScene = TITLE in main — keep it; it seeds the manager's first transition check correctly.
