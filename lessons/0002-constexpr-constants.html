Lesson 02 · Refactor

Give your constants a type

The #define wall at the top of scene_gameplay.cpp is text substitution with no type, no scope, no safety. constexpr fixes all three.

What #define actually does

A macro is not a constant. It's a find-and-replace the preprocessor runs before the compiler ever sees your code. #define ENEMY_SPEED 50.0f means "paste the text 50.0f everywhere the letters ENEMY_SPEED appear." The compiler never knows a thing called ENEMY_SPEED existed.

That causes three concrete problems, all present in your file:

1 — No type

#define ENEMY_SPEED 50.0f      // is this float? double? the compiler decides per-use
#define MAX_ENEMIES_PER_ROW 11 // int

Look at line 143: int speed = state->enemySpeed; — and enemySpeed is declared int in your struct, even though ENEMY_SPEED is 50.0f. The float silently truncates. A typed constant makes the mismatch visible (and fixable).

2 — No scope

Macros ignore namespaces, classes, functions — they leak everywhere in the translation unit after their #define. Two files each defining ROW_PADDING differently is a silent landmine. A constexpr obeys normal scope rules like any variable.

3 — Invisible to the debugger & error messages

Because the name is gone before compilation, your debugger can't show ENEMY_SPEED, and a bad expression reports the substituted text, not the name you wrote.

Why this fits the mission

"Better-structured" means the compiler understands your intent, not just the preprocessor. Typed constants are the groundwork for the entity classes coming later — those classes will hold these values as real typed members, and you can't build clean members on top of untyped text macros.

The tool: constexpr

constexpr = "a real, typed, named value the compiler computes at compile time." Same zero runtime cost as a macro, but it's a genuine variable: it has a type, obeys scope, shows up in the debugger, and participates in overload resolution and templates. [cppreference: constexpr, Core Guidelines ES.31: don't use macros for constants]

- #define PLAYER_RADIUS 12.5f
+ constexpr float PLAYER_RADIUS = 12.5f;

- #define MAX_ENEMIES_PER_ROW 11
+ constexpr int MAX_ENEMIES_PER_ROW = 11;

- #define ENEMY_SPEED 50.0f
+ constexpr float ENEMY_SPEED = 50.0f;

Read it as: constexpr (compile-time constant) float (the type) PLAYER_RADIUS (the name) = 12.5f (the value). The type is the thing the macro never had.

Your task feedback loop

  1. Convert the simple scalar macros (lines 13–15, 19–24) to constexpr with the right type: float for the .0f values, int for the counts/paddings.
  2. Build. Then look hard at int speed = state->enemySpeed; (line ~143) and int enemySpeed; in the struct. Now that ENEMY_SPEED is openly float, does enemySpeed want to be float too? Make that call and justify it.
  3. Leave the function-like macros for now: CANVAS_OFFSET, INITIAL_PLAYER_POS, START_GRID_POS. They build a Vector2 and reference windowSize — see the trap below.
  4. Leave FOR_EACH_PROJECTILE / FOR_EACH_ENEMY entirely — those are lesson 3.
The trap to discover: try converting INITIAL_PLAYER_POS to constexpr Vector2. It references windowSize, which is declared extern const Vector2 windowSize; — defined in another file, value not known at compile time here. A constexpr needs its value known at compile time. What does the compiler say? What keyword could you use instead for a typed constant whose value is set at runtime? (Hint: drop constexpr, keep const.) Bring me your answer.
