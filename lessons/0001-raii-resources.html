Lesson 01 · Refactor

Make resources unload themselves

Your first true C++ idiom: RAII. We delete UnloadGameResources and never worry about it again.

The C habit we're breaking

Right now main.cpp owns the textures and sounds manually. You load them, and you must remember to unload them — by hand, in the right order, at the right time:

// main.cpp — the C way: two functions that must stay in sync
void LoadGameResources(GameResources *r) {
  r->playerTexture = LoadTexture("player.png");
  /* ...four more... */
}
void UnloadGameResources(GameResources *r) {
  UnloadTexture(r->playerTexture);
  /* ...four more... */
}

int main() {
  GameState state = {};
  LoadGameResources(&state.resources);   // acquire
  /* ... game loop ... */
  UnloadGameResources(&state.resources); // release — easy to forget, or to run too late/early
}

Every Load* needs a matching Unload*, and the compiler does nothing to enforce it. Add an early return or an exception, and the cleanup is silently skipped — a leak. This is exactly the class of bug C++ was designed to delete.

The idea: RAII

Resource Acquisition Is Initialization. Tie a resource's lifetime to an object's lifetime. You acquire in the constructor; you release in the destructor. When the object goes out of scope, C++ guarantees the destructor runs — normal exit, early return, or exception. Cleanup becomes impossible to forget. [cppreference: RAII, Core Guidelines R.1]

Why this fits the mission

A "better-structured game" means structure that can't be misused. After this refactor it is literally impossible to leak these resources or call cleanup in the wrong order — the language enforces it, not your discipline. Smallest change, biggest safety win: that's why it's lesson one.

The refactor

Step 1 — turn the struct into a class with a constructor + destructor

In include/game.h, replace the plain GameResources struct:

// game.h
struct GameResources {
  Texture2D playerTexture;
  Texture2D enemyTexture;
  Texture2D laserTexture;
  Sound laserSound;
  Sound explosionSound;

  GameResources();             // constructor: acquires (was LoadGameResources)
  ~GameResources();            // destructor: releases  (was UnloadGameResources)

  // An OpenGL texture handle must not be copied — two copies would each
  // try to unload the same GPU handle. Forbid copying outright:
  GameResources(const GameResources&) = delete;
  GameResources& operator=(const GameResources&) = delete;
};
The two = delete lines are the rule of three/five in action: once a class owns a raw handle, the default copy is wrong, so you delete it. We'll add move later if a scene ever needs to hand resources off.

Step 2 — move the load/unload bodies into the special members

In main.cpp, delete the two free functions and their declarations in game.h, and write instead:

// main.cpp
GameResources::GameResources() {
  playerTexture  = LoadTexture("player.png");
  enemyTexture   = LoadTexture("enemy.png");
  laserTexture   = LoadTexture("laser.png");
  laserSound     = LoadSound("laser.wav");
  explosionSound = LoadSound("explosion.wav");
}

GameResources::~GameResources() {
  UnloadTexture(playerTexture);
  UnloadTexture(enemyTexture);
  UnloadTexture(laserTexture);
  UnloadSound(laserSound);
  UnloadSound(explosionSound);
}

Step 3 — delete the manual calls

GameResources lives inside GameState state, so its constructor and destructor now fire automatically when state is born and dies. Remove both manual calls from main():

int main() {
  /* ...InitWindow, InitAudioDevice... */
  GameState state = {};            // GameResources ctor runs here -> textures load
  - LoadGameResources(&state.resources);
  InitGameplay(&state);
  /* ... game loop ... */
  - UnloadGameResources(&state.resources);
  CloseAudioDevice();
  CloseWindow();
}                                  // ~GameResources runs at scope exit -> textures unload
Order trap (worth knowing): textures must be unloaded before CloseWindow()/CloseAudioDevice() close the GL/audio context. Because state is a local declared before those calls, and locals destruct in reverse order at the closing } — its destructor would run after CloseWindow(). Wrap the game in an inner scope so state dies first:

int main() {
  InitWindow(...); InitAudioDevice();
  {
    GameState state = {};
    /* load + loop */
  }  // ~GameResources here — GL/audio still alive. Good.
  CloseAudioDevice();
  CloseWindow();
}

Build & verify feedback loop

  1. Run make (or your usual build). It should compile with no warnings.
  2. Launch the game. Sprites render exactly as before — same behaviour, less code.
  3. Confirm UnloadGameResources no longer exists anywhere: grep -rn UnloadGameResources src include returns nothing.
  4. You deleted ~20 lines and an entire class of bug. That's the win.
