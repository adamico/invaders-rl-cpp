Lesson 12 · Refactor · Invariant ownership

Whoever owns the data owns the rule

You spotted it yourself: the collision functions smell. Not because they're free functions — that part is right — but because they reach into the swarm's guts and maintain an invariant that isn't theirs to maintain. This lesson fixes the deepest of those smells.

The bug waiting to happen

Look at one line inside CheckBulletEnemyCollisions:

enemy.active = false;
state.swarm.activeCount--;   // ← a stranger maintaining the swarm's invariant

The Swarm has an invariant: activeCount equals the number of enemies whose active flag is true. Right now that rule is enforced outside the swarm, in collision code. Two things must happen together — flip the flag, decrement the count — and nothing guarantees they do. The day someone deactivates an enemy somewhere else and forgets the --, activeCount drifts, and your victory check (activeCount <= 0) fires early or never. It will compile fine and fail silently — the worst kind. [Core Guidelines C.2: use class (not struct) when there's an invariant; the class's job is to keep it]

The principle: an invariant lives with the data it constrains

If activeCount must always match the active flags, then the only code allowed to change either of them should be the Swarm itself. That way the rule is enforced in one place and cannot be violated from outside. This is the same idea as SceneManager owning the current_/shown_ pair (lesson 7) — you've met it before; now you're applying it to catch your own smell.

The fix: a verb on the owner

Give the swarm an operation that maintains both fields atomically, then make it the only path:

// swarm.h
struct Swarm {
  ...
  void deactivate(Enemy& enemy);   // the one sanctioned way to kill an enemy
};
// swarm.cpp
void Swarm::deactivate(Enemy& enemy) {
  if (!enemy.active) return;       // idempotent guard — the count can't drift
  enemy.active = false;
  activeCount--;
}

The guard matters: it makes deactivate safe to call twice on the same enemy without double-decrementing. The invariant is now structurally protected — there's no way to flip the flag without adjusting the count, because the only code that touches them lives here.

Tell, don't ask

Now the collision function stops asking for the swarm's internals and mutating them, and instead tells the swarm what happened:

// scene_gameplay.cpp — CheckBulletEnemyCollisions
if (CheckCollisionCircles(bullet.pos, bullet.radius, enemy.pos, enemy.radius)) {
  PlaySound(state.resources.explosionSound);
  bullet.deactivate();              // the projectile owns its own flag
  state.swarm.deactivate(enemy);    // the swarm owns flag + count, together
  state.score += enemy.scoreValue;
}
// CheckPlayerEnemyCollisions — same move
if (CheckCollisionCircles(state.player.pos, state.player.radius, enemy.pos, enemy.radius)) {
  state.swarm.deactivate(enemy);
  state.player.health = 0;
}

Tell, Don't Ask: don't pull an object's data out to make a decision and write it back — tell the object to do the thing, and let it manage its own state. state.swarm.activeCount-- reaches two levels into another object's guts (a Law of Demeter violation); swarm.deactivate(enemy) respects the boundary. The collision code now reads as intent — "a bullet hit an enemy: kill the bullet, kill the enemy, score it" — not as bookkeeping. [Fowler — TellDontAsk; Law of Demeter]

// game.h — Projectile gains the same intent-revealing verb (it has no count, so it's trivial — but symmetric)
struct Projectile {
  ...
  void deactivate() { active = false; }
};

Why deactivate is a member but the collision check stays free

No contradiction with lessons 9–11. The rule is C.4: a function is a member when it needs to maintain the type's representation/invariant. deactivate maintains the swarm's count invariant → member. The collision check mediates two unrelated types and maintains neither's invariant → free. You're applying the same rule to reach opposite conclusions for different functions. That's the skill: not "members vs free" as dogma, but "who owns the invariant being touched?"

Smells we're NOT fixing today (named, deferred on purpose)

  • The name lies. CheckBulletEnemyCollisions doesn't just check — it scores, plays sound, kills. Splitting detection (returns hits) from effect (applies them) would make it testable and honest. Worth doing, but it's a separate idea; one concept per lesson.
  • Duplicated if (!active) continue; in every loop. That repetition hints at an "iterate only active elements" helper on the pool/swarm. Note it; don't chase it yet.

Recognising and deferring smells is a real skill — you don't have to fix everything you see in one pass.

Files you'll touch (checklist)

  1. include/swarm.h — declare void deactivate(Enemy& enemy);.
  2. src/swarm.cpp — define it, with the idempotent if (!enemy.active) return; guard.
  3. include/game.h — add void deactivate() { active = false; } to Projectile (inline is fine; it's trivial).
  4. src/scene_gameplay.cpp — in both collision functions, replace the manual flag/count pokes with state.swarm.deactivate(enemy) and bullet.deactivate(). Delete every activeCount-- outside the swarm.
  5. Build & play: shoot enemies → score + count drop correctly → clear them all → victory still fires exactly once. Take a hit → death.
Traps: (1) Grep for any remaining activeCount outside swarm.cpp — there should be exactly zero writes left. (2) Don't forget the idempotent guard; without it, a future double-kill silently corrupts the count. (3) Victory check stays in the scene (activeCount <= 0 is a read, which is fine — reads don't threaten the invariant).
