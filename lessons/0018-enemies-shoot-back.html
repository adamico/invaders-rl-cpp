Lesson 18 · Gameplay · Enemies shoot back

Finish the game: make the swarm fight back

The refactor arc is done — the code is clean. Now we build on it. First missing piece of a real Space Invaders: enemies that shoot. You already have everything needed — a generic ProjectilePool, a fire(from, dir) hook that's sat unused since L10, and health = 5 that currently means nothing. Today they all pay off.

The plan, and why each piece is something you already own

This is the whole point of the refactor: new gameplay is now composition of existing parts, not new plumbing. The generic ProjectilePool earns its keep the moment you need a second one.

Step 1 — a second projectile pool + a fire timer (state)

Enemy bullets must NOT share the player's pool — the bullet/enemy collision loop would then treat enemy bullets as enemy-killers. Separate pools keep the two populations cleanly apart. Add to GameState:

// game_state.h
struct GameState {
  Player player;
  ProjectilePool projectilePool;       // player bullets (up)
  ProjectilePool enemyProjectilePool;  // NEW: enemy bullets (down)
  Swarm swarm;
  bool victory;
  GameResources resources;
  int score;
  float enemyFireCooldown;             // NEW: seconds until next enemy shot
};

The same class, instantiated twice — zero new code in projectile_pool.h. That reuse is the dividend of making it generic instead of bespoke.

Step 2 — reset the new state

In initGameplay, mirror what you do for the player pool and seed the timer:

// scene_gameplay.cpp — initGameplay()
state->projectilePool.reset();
state->enemyProjectilePool.reset();   // NEW
state->enemyFireCooldown = ENEMY_FIRE_INTERVAL;  // NEW

Add the tuning constants near the other gameplay constants at the top of the file:

constexpr float ENEMY_FIRE_INTERVAL = 0.8f;  // avg seconds between shots
constexpr float ENEMY_PROJECTILE_SPEED = 350.0f;  // (optional, see note)
Note on speed: Projectile::spawn hard-codes speed = PROJECTILE_SPEED (500). Enemy bullets reusing spawn will move at that same speed — fine for v1. If you want slower enemy shots later, that's a real design question (parameterise spawn's speed, or give the pool a speed). Don't solve it now; ship the simple version first. Flagged, not fixed.

Step 3 — the firing logic (timer + reservoir pick)

A new free function, in the same C-style-free-function spirit as the collision helpers (C.4 — it orchestrates several objects, belongs to none). Two ideas here, both worth understanding:

The accumulator/cooldown pattern

You can't fire "every 0.8s" by counting frames — frame time varies. Instead subtract elapsed time each frame; when the bank hits zero, act and refill. This is the canonical game-timer pattern:

Reservoir sampling — pick one active enemy uniformly, in one pass

Swarm hides its pool, but it's iterable (the begin/end you added in L16). You don't know how many are active without looking, and you don't want two passes. Reservoir sampling picks a uniformly-random item from a stream of unknown length in a single pass: keep the k-th item seen with probability 1/k, and the last keeper is uniform over all. For one item:

// scene_gameplay.cpp
void updateEnemyFire(GameState& state, float dt) {
  state.enemyFireCooldown -= dt;
  if (state.enemyFireCooldown > 0.0f) return;        // not yet

  // reservoir-pick one active enemy in a single pass
  Enemy* shooter = nullptr;
  int seen = 0;
  for (Enemy& enemy : state.swarm) {
    if (!enemy.active) continue;
    ++seen;
    if (GetRandomValue(1, seen) == 1) shooter = &enemy;  // keep with prob 1/seen
  }

  if (shooter) {
    state.enemyProjectilePool.fire(shooter->pos, {0.0f, 1.0f});  // downward
    PlaySound(state.resources.laserSound);
  }
  state.enemyFireCooldown = ENEMY_FIRE_INTERVAL;       // refill the bank
}

GetRandomValue(min, max) is raylib's inclusive integer RNG. GetRandomValue(1, seen) == 1 is true with probability exactly 1/seen — that's the reservoir trick in one line. [Reservoir sampling · raylib GetRandomValue]

Step 4 — enemy bullets damage the player

A second collision helper, mirroring detectBulletEnemyCollisions but pointing the other way. This is where health finally becomes a real resource instead of a 0/1 flag:

// scene_gameplay.cpp
void detectEnemyBulletPlayerCollisions(GameState& state) {
  for (Projectile& bullet : state.enemyProjectilePool) {
    if (!bullet.active) continue;

    if (overlaps(bullet, state.player)) {
      bullet.deactivate();
      state.player.health -= 1;        // 5 hits, not instant death
    }
  }
}

overlaps is symmetric and templated (L17 made Player satisfy Collidable already — it has pos + radius), so overlaps(bullet, state.player) just works with no new code.

Step 5 — wire it into the frame

Update and draw the new pool alongside the old one. In SceneGameplay::update:

std::unique_ptr<Scene> SceneGameplay::update(GameState& state, float dt) {
  if (state.swarm.update(dt)) state.player.health = 0;
  state.player.update(dt);
  playerShoot(state);
  updateEnemyFire(state, dt);                  // NEW
  state.projectilePool.update(dt);
  state.enemyProjectilePool.update(dt);        // NEW
  detectBulletEnemyCollisions(state);
  detectPlayerEnemyCollisions(state);
  detectEnemyBulletPlayerCollisions(state);    // NEW

  if (state.swarm.activeCount() <= 0) state.victory = true;
  if (state.victory || state.player.health <= 0)
    return std::make_unique<SceneGameover>();

  return nullptr;
}

And in SceneGameplay::draw, render the enemy bullets (reuse the laser texture):

state.projectilePool.draw(state.resources.laserTexture);
state.enemyProjectilePool.draw(state.resources.laserTexture);  // NEW

Files you'll touch (checklist)

  1. include/game_state.h — add ProjectilePool enemyProjectilePool; and float enemyFireCooldown;.
  2. src/scene_gameplay.cpp — add the two constants; reset the pool + seed the timer in initGameplay; add updateEnemyFire and detectEnemyBulletPlayerCollisions; call all three new lines in update; draw the new pool.
  3. Build & play: enemies fire downward; bullets chip health one per hit (HUD shows it drop 5→0); five hits = gameover. Behaviour of player shooting unchanged.
Traps: (1) Don't reuse the player's pool for enemy bullets — separate instance, or your bullet/enemy collision loop will let enemy fire kill enemies. (2) Decrement health -= 1 on the enemy-bullet hit, but remember detectPlayerEnemyCollisions still sets health = 0 on contact (a body-block is still instant death — that's fine for now; "real lives/respawn" is a later lesson). (3) GetRandomValue(1, seen) — the lower bound is 1, not 0; == 1 gives probability 1/seen. (4) Seed the cooldown in initGameplay or the first frame fires immediately (cooldown starts at 0/garbage).
