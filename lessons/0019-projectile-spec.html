Lesson 19 · Design · Parameter object

Stop threading arguments: give each pool a profile

You felt it yourself: enemy bullets differ from player bullets by speed, then direction, then a flipped sprite — and each difference becomes another argument bolted onto fire() and spawn(), at every call site, forever. That's a known smell with a known fix. The traits aren't per-shot; they're per-weapon. So the pool should own them.

Name the smell

Right now your signatures have been growing:

// the telescoping you already have
bool fire(Vector2 from, Vector2 dir = {0,-1}, float speed = 500.0f);
void spawn(Vector2 from, Vector2 dir = {0,-1}, float speed = 500.0f);

Add "flip the sprite for enemy bullets" and it's a fourth parameter; add damage, sound, sprite and it's the seventh. This is the Long Parameter List / Data Clump smell, and the fix is Introduce Parameter Object: the cluster of values that always travel together becomes one named thing. [Fowler — Introduce Parameter Object]

The key insight — ownership, not just bundling. Bundling the args into a struct you still pass on every fire() call would help a little. But notice who those values belong to: speed, direction, flip are constant for the whole pool's lifetime — every player bullet is identical; every enemy bullet is identical. They're not properties of a shot, they're properties of the weapon. So the pool should own the profile and set it once. Then fire() shrinks to the one thing that really does vary per shot: where.

Step 1 — the spec struct (with default member initializers)

A plain aggregate of the traits, with sane defaults. Put it in game.h above Projectile (the projectile needs it for spawn):

// game.h — above struct Projectile
struct ProjectileSpec {
  float speed = 500.0f;
  Vector2 dir = {0.0f, -1.0f};
  bool flipV  = false;        // render the sprite upside-down (enemy bullets)
};

Those = ... are default member initializers: any field you don't mention at construction gets the default. Combined with C++20 designated initializers (which you unlocked in L17), configuring a pool reads like a labelled record — you set only what differs:

state->projectilePool.reset({.speed = 500.0f, .dir = {0, -1}});            // flipV defaults false
state->enemyProjectilePool.reset({.speed = 350.0f, .dir = {0, 1}, .flipV = true});

Step 2 — the pool owns one spec, set in reset

fire loses all its trait parameters — back down to just the position:

// projectile_pool.h
struct ProjectilePool {
  void reset(ProjectileSpec spec);   // configure + clear
  void update(float deltaTime);
  void draw(const Texture2D& texture) const;
  bool fire(Vector2 from);           // position only — pool knows the rest
  int activeCount() const { return pool.activeCount(); }
  // begin/end unchanged...
private:
  Pool<Projectile, MAX_PROJECTILES> pool;
  ProjectileSpec spec{};             // the weapon profile
};
// projectile_pool.cpp
void ProjectilePool::reset(ProjectileSpec newSpec) {
  spec = newSpec;
  for (auto& projectile : pool)
    projectile = {.active = false};
}

bool ProjectilePool::fire(Vector2 from) {
  Projectile* projectile = pool.acquire();
  if (!projectile) return false;
  projectile->spawn(from, spec);     // hand the slot the whole profile
  return true;
}

Step 3 — spawn takes the spec, not loose args

// game.h
struct Projectile {
  Vector2 pos{};
  Vector2 dir{};
  float speed{};
  float radius{};
  bool active{};
  bool flipV{};                      // NEW: carried for draw()

  void spawn(Vector2 from, const ProjectileSpec& spec);   // const ref: read-only, no copy
  void update(float deltaTime);
  void draw(const Texture2D& texture) const;
  void deactivate() { active = false; }
};
// scene_gameplay.cpp
void Projectile::spawn(Vector2 from, const ProjectileSpec& spec) {
  pos    = from;
  dir    = spec.dir;
  speed  = spec.speed;
  flipV  = spec.flipV;
  radius = PROJECTILE_RADIUS;
  active = true;
}

Note const ProjectileSpec& — read-only access, no copy (you only read the fields). The spec flows pool → spawn → projectile fields; after that the projectile is self-sufficient.

Step 4 — fix the directional retire bug, and flip the sprite

Your update currently retires a bullet with active = pos.y > 0 — correct for upward player bullets, but a downward enemy bullet keeps pos.y > 0 forever after it leaves the bottom of the screen: it never deactivates, silently hogging a pool slot. Retire on either edge:

void Projectile::update(float dt) {
  if (!active) return;
  pos = Vector2Add(pos, Vector2Scale(dir, speed * dt));
  active = pos.y > 0 && pos.y < GetScreenHeight();   // off top OR bottom = dead
}

And the raylib bit — draw the sprite flipped when flipV. A texture is flipped vertically by giving DrawTextureRec a source rectangle with negative height:

void Projectile::draw(const Texture2D& texture) const {
  if (!active) return;
  Rectangle src = {0, 0, (float)texture.width,
                   flipV ? -(float)texture.height : (float)texture.height};
  Vector2 drawPos = Vector2Add(pos, CANVAS_OFFSET);
  DrawTextureRec(texture, src, drawPos, WHITE);
  DrawCircleLinesV(pos, radius, RED);
}

(Negative source height is raylib's idiom for a vertical flip — it samples the rows bottom-to-top. [raylib DrawTextureRec])

Step 5 — update the call sites (they get simpler)

// initGameplay — configure each pool once
state->projectilePool.reset({.speed = 500.0f, .dir = {0.0f, -1.0f}});
state->enemyProjectilePool.reset({.speed = 350.0f, .dir = {0.0f, 1.0f}, .flipV = true});

// playerShoot — position only
if (state.projectilePool.fire(state.player.pos)) PlaySound(state.resources.laserSound);

// updateEnemyFire — position only
state.enemyProjectilePool.fire(shooter->pos);

The payoff to feel: a future fourth trait (damage, sprite, sound) is one field in ProjectileSpec + one default — and zero call sites change. The interface stopped growing.

Files you'll touch (checklist)

  1. include/game.h — add struct ProjectileSpec (above Projectile); add bool flipV{}; to Projectile; change spawn to spawn(Vector2 from, const ProjectileSpec&).
  2. include/projectile_pool.h — reset(ProjectileSpec); fire(Vector2 from) (drop dir/speed); add private ProjectileSpec spec{};.
  3. src/projectile_pool.cpp — store the spec in reset; fire calls spawn(from, spec).
  4. src/scene_gameplay.cpp — rewrite Projectile::spawn/update/draw per above; configure both pools in initGameplay; simplify the two fire call sites.
  5. Build & play: player bullets up at 500, enemy bullets down at 350 with a flipped sprite; both retire off-screen; fire() takes only a position again.
Traps: (1) ProjectileSpec must be declared before Projectile in game.h (spawn's signature names it). (2) Designated initializers require fields in declaration order and C++20 (you have it) — {.dir=…, .speed=…} in the wrong order won't compile. (3) Don't forget to actually store the spec in reset before clearing — a pool that never set its spec fires with default-constructed traits. (4) The flip is a source-rect negative height, not a negative draw position.
