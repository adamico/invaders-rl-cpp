Lesson 20 · Design · Tell, don't ask (round 2)

Put behaviour where the data lives — when there's a rule to protect

You left yourself five // REFACTOR breadcrumbs, and they all say the same thing: the scene reaches in and pokes raw fields — player.health = 0, health -= 1, an opaque swarm.update() bool. The fix is to move the behaviour onto the object that owns the data. But the real lesson is the judgment: which pokes deserve a method, and which (like score) are fine as they are.

The principle, sharpened

"Tell, Don't Ask" (you met it in L12) says: don't pull an object's data out to make a decision the object could make itself. But a wrapper for its own sake is just noise (L11 — when not to make a class). The sharp version:

Behaviour belongs with data when there's an invariant or rule to protect. player.health has rules — it must never go below zero, "dead" means a specific thing, and there should be exactly one definition of taking damage. Those rules deserve methods that own them. A bare score counter has no rule — adding points is just +=. Wrapping it buys nothing. Same reasoning, opposite conclusions. That contrast IS this lesson.

Part 1 — Player owns its health (the one with rules)

Three behaviours, then hide the field behind them. In game.h:

// game.h
struct Player {
  Vector2 pos{};
  Vector2 dir{};
  float speed{};
  float radius{};

  void reset(Vector2 startPos);            // NEW: replaces the aggregate init (see note)
  void update(float deltaTime);
  void draw(const Texture2D& texture) const;

  void takeDamage(int amount = 1);         // NEW
  void die();                              // NEW
  bool isAlive() const { return hp > 0; } // NEW
  int  health() const { return hp; }       // NEW: read-only query for the HUD

private:
  int hp{};                                // was public `int health` — now behind methods
};
// scene_gameplay.cpp
void Player::takeDamage(int amount) {
  hp -= amount;
  if (hp < 0) hp = 0;        // the rule the caller no longer has to know
}

void Player::die() { hp = 0; }

takeDamage(int amount = 1) uses a default argument — call takeDamage() for the common 1-point hit, takeDamage(3) later for a heavy enemy. The clamp-at-zero rule lives in exactly one place now.

Why a reset() appears. Making hp private has a consequence: a class with a private data member is no longer an aggregate, so the (Player){.pos = …, .health = …} designated-initializer in initPlayer stops compiling. That's not a wart — it's the same shape as Swarm::reset() and ProjectilePool::reset(): once a type has private invariants, it owns its setup. Move the init logic in: 
// scene_gameplay.cpp — Player owns its own setup
void Player::reset(Vector2 startPos) {
  pos = startPos;
  dir = {0.0f, 0.0f};
  speed = PLAYER_SPEED;
  radius = PLAYER_RADIUS;
  hp = PLAYER_HEALTH;
}

// initGameplay — the free initPlayer() collapses into this
Vector2 startPos = {GetScreenWidth() / 2.0f, GetScreenHeight() - (PLAYER_RADIUS * 4)};
state->player.reset(startPos);

Part 2 — the collision sites now tell

Your three breadcrumbs become method calls. The scene stops knowing how health works:

// detectPlayerEnemyCollisions — body-block = instant death
state.player.die();                  // was: state.player.health = 0;

// detectEnemyBulletPlayerCollisions — one bullet, one point
state.player.takeDamage();           // was: state.player.health -= 1;

And the HUD reads through the query: drawHud(state.player.health(), state.score);.

Part 3 — clarify the opaque swarm.update() bool

Your comment: "the return value of swarm.update() is unclear." It returns a bare bool that the call site has to remember means "breached." Split the command from the query — update advances the swarm (a command, returns nothing); a named query answers the question:

// swarm.h
void update(float deltaTime);     // was: bool update(...)
bool hasBreached() const { return breached; }   // NEW: intent-revealing
private:
  bool breached{};                // NEW: set during update()

In Swarm::update, drop bool breached as a local and set the member instead (reset it to false at the top of each update; remove the return). The scene reads like prose:

state.swarm.update(dt);
if (state.swarm.hasBreached()) state.player.die();

This is Command-Query Separation: a method either does something or answers something, not both-via-a-cryptic-return. [Fowler — CommandQuerySeparation]

Part 4 — selection belongs on the Swarm (and std::optional)

You spotted updateEnemyFire doing two jobs: finding a shooter and firing. The finding reaches into the swarm's enemies — but the swarm owns the enemies; it should answer "where would a shot come from?" Move the reservoir pick onto Swarm, and return only what the caller needs — a position, or nothing:

// swarm.h
#include <optional>
...
std::optional<Vector2> randomShooterPos() const;   // NEW
// swarm.cpp — the reservoir loop moves here, behind the encapsulation
std::optional<Vector2> Swarm::randomShooterPos() const {
  const Enemy* shooter = nullptr;
  int seen = 0;
  for (const Enemy& enemy : pool) {
    if (!enemy.active) continue;
    ++seen;
    if (GetRandomValue(1, seen) == 1) shooter = &enemy;
  }
  if (!shooter) return std::nullopt;     // "no active enemies"
  return shooter->pos;
}

std::optional<Vector2> is "a Vector2, or nothing" expressed in the type — no sentinel value, no out-param, no leaking an Enemy* to the caller (the representation stays hidden, as in L16). updateEnemyFire shrinks to its one real job:

void updateEnemyFire(GameState& state, float dt) {
  state.enemyFireCooldown -= dt;
  if (state.enemyFireCooldown > 0.0f) return;

  if (auto pos = state.swarm.randomShooterPos())   // has_value? then *pos is valid
    state.enemyProjectilePool.fire(*pos);

  state.enemyFireCooldown = ENEMY_FIRE_COOLDOWN;
}

if (auto pos = …) tests the optional's truthiness; *pos unwraps it. Firing logic here, selection logic in the swarm — each does one thing. [cppreference — std::optional]

Part 5 — the one you DON'T do: score stays an int

Your // REFACTOR: should be state.score.add(...) is a fair instinct toward symmetry — but follow the rule from the top of the lesson. Is there an invariant on score? No: it only ever goes up, by +=, with no clamp, no derived state, no "you can't do that" rule. A Score::add() wrapping a single += is a trivial getter/setter in disguise — the exact thing Core Guidelines C.131 warns against. So leave it:

state.score += enemy.scoreValue;     // deliberately unchanged — no rule to protect

Knowing when not to add structure is the same skill as knowing when to. Revisit it the day score grows a rule — a combo multiplier, or a persisted high-score. Then it earns a class. Not before. (YAGNI.)

Files you'll touch (checklist)

  1. include/game.h — Player: hp private; add reset, takeDamage(int=1), die, isAlive, health().
  2. include/swarm.h — #include <optional>; update returns void; add hasBreached() const + private bool breached{}; declare randomShooterPos() const.
  3. src/scene_gameplay.cpp — define Player::reset/takeDamage/die; fold initPlayer into player.reset(); collision sites call die()/takeDamage(); HUD uses health(); updateEnemyFire uses randomShooterPos(); update() uses hasBreached(); leave score += alone.
  4. src/swarm.cpp — set member breached in update (reset to false at top, no return); move the reservoir loop into randomShooterPos.
  5. Build & play: identical behaviour; the scene now reads as intent (die(), takeDamage(), hasBreached()), hp is unreachable from outside, and selection lives in the swarm.
Traps: (1) Privatising hp breaks the designated-init in initPlayer — that's expected; the reset() method replaces it. (2) Remove the return breached; AND the local bool breached in update, else the member is shadowed and hasBreached() always reads false. (3) Don't deref an empty optional — gate with if (auto pos = …) before *pos. (4) randomShooterPos is const, so iterate const Enemy& and hold a const Enemy*. (5) Resist wrapping score — the lesson is that NOT every poke becomes a method.
