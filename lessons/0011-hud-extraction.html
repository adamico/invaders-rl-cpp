Lesson 11 · Refactor · When NOT to make a class

The HUD is a responsibility, not an object

You've extracted two classes in a row. The reflex now is "HUD → make a HudClass." Resist it. The HUD owns no data — it's pure rendering. The skill this lesson builds is recognising when a responsibility wants a function, not a class.

The smell, and the trap

SceneGameplay::draw mixes three different jobs:

void SceneGameplay::draw(const GameState& state) const {
  // (1) frame lifecycle
  BeginDrawing();
  ClearBackground(BLACK);
  // (2) world rendering
  state.player.draw(state.resources.playerTexture);
  state.swarm.draw(state.resources.enemyTexture);
  state.projectilePool.draw(state.resources.laserTexture);
  // (3) HUD / overlay — the bit to extract
  DrawText(TextFormat("Health: %i", state.player.health), 20, GetScreenHeight()-40, 20, WHITE);
  DrawText(TextFormat("Score: %i", state.score), 20, 20, 20, WHITE);
  EndDrawing();
}

The HUD lines are a separate concern (UI overlay) buried in the scene's draw. Extract them — but here's the trap: Swarm and ProjectilePool earned classes because they owned mutable data (arrays, counters, an invariant to protect). The HUD owns nothing. It reads two numbers and draws text. Wrapping it in a class would invent a stateless object whose only method takes all its inputs as arguments — a class pretending to be a function. [Core Guidelines C.4: make a function a member only if it needs direct access to the representation of a class]

The test: does it have state to protect?

A class is justified when there's data + invariants to keep consistent (the pool's active flags, the swarm's direction/count). If a thing is "take some inputs, produce some output/side-effect, keep nothing," it's a free function. The HUD keeps nothing between frames. So: free function. This is the same C.4 rule you used to keep collisions free in lessons 9–10 — now applied to decide a class shouldn't exist at all. SRP means "one job per unit," not "everything is an object."

Extract it — and decouple while you're there

Two ways to pass the data in:

// Option 1 — pass the whole state (convenient, but couples HUD to GameState)
void drawHud(const GameState& state);

// Option 2 — pass only what it draws (HUD knows nothing about GameState)  ← prefer
void drawHud(int score, int health);

Prefer option 2. The HUD's job is "render these numbers on screen" — it shouldn't need to know a GameState type exists. Passing primitives means the HUD has zero dependencies on the rest of the game: you could reuse it, test it, or call it from a different scene with made-up values. That's looser coupling for free. (If the HUD later grows to a dozen fields, revisit — at that point a small HudData struct or const GameState& may read better. For two ints, primitives win.) [Core Guidelines I.5/F.2: a function should take only what it needs]

// NEW FILE: include/hud.h
#pragma once
void drawHud(int score, int health);
// NEW FILE: src/hud.cpp
#include "hud.h"
#include "raylib.h"

constexpr int HUD_FONT_SIZE = 20;
constexpr int HUD_MARGIN = 20;

void drawHud(int score, int health) {
  DrawText(TextFormat("Score: %i", score), HUD_MARGIN, HUD_MARGIN, HUD_FONT_SIZE, WHITE);
  DrawText(TextFormat("Health: %i", health), HUD_MARGIN,
           GetScreenHeight() - 2 * HUD_MARGIN, HUD_FONT_SIZE, WHITE);
}

Notice hud.h includes nothing game-specific — not even game_state.h. That's the payoff of option 2.

Where the boundary falls: HUD draws, the scene owns the frame

drawHud does not call BeginDrawing/EndDrawing or ClearBackground. Those bracket the entire frame and define draw order — that's the scene's lifecycle responsibility, not the HUD's. The HUD only emits its text into the already-open frame. Getting this boundary right is the actual lesson: an extracted unit should own its slice and nothing of its caller's job. If drawHud called BeginDrawing, you couldn't compose it with the world rendering in one frame.

Rewire the scene

// scene_gameplay.cpp — add #include "hud.h"
void SceneGameplay::draw(const GameState& state) const {
  BeginDrawing();
  ClearBackground(BLACK);

  state.player.draw(state.resources.playerTexture);
  state.swarm.draw(state.resources.enemyTexture);
  state.projectilePool.draw(state.resources.laserTexture);

  drawHud(state.score, state.player.health);   // was the two DrawText lines + font_size
  EndDrawing();
}

The scene's draw now reads as exactly its three jobs: open frame → render world → render overlay → close frame. Each line is one responsibility.

Files you'll touch (checklist)

  1. include/hud.h (new) — just void drawHud(int score, int health);, no game includes.
  2. src/hud.cpp (new) — the two DrawText calls + HUD layout constants.
  3. src/scene_gameplay.cpp — #include "hud.h"; replace the two DrawText lines (and the local font_size) with one drawHud(...) call.
  4. No Makefile change — src/*.cpp globbed.
  5. Build & play: score + health still render in the same spots; numbers update as you score / take damage.
Traps: (1) Don't drag BeginDrawing/EndDrawing into drawHud — leave frame lifecycle in the scene. (2) Don't #include "game_state.h" in hud.h — if you feel the urge, you've slipped back to option 1. (3) The old local int font_size is now a constant in hud.cpp; remove the stray one from the scene.
